# 機能仕様: WakeOrPay（寝坊防止ペナルティアラーム）

**Feature Branch**: `001-wake-or-pay`
**Created**: 2026-04-09
**Status**: Draft
**Input**: ユーザー説明: "寝坊防止のペナルティアラームアプリ「WakeOrPay」。ユーザーは寝る前にアラーム時刻をブラウザ上でセットする。セットした時刻までにブラウザ上で解除ボタンを押せば何も起きない。解除しないまま時刻を過ぎると、OpenAI APIが1回自動実行され、ユーザーのAPIキーに課金が発生する（罰金）。MVP段階では自分一人が使う想定。認証不要。"

## Clarifications

### Session 2026-04-09

- Q: ペナルティ発火のスケジューラはどこで動くか？ → A: Upstash QStash で指定時刻に `/api/punish` を1回叩くスケジュールを登録する。解除時は QStash のスケジュールID でキャンセルする。スケジュールIDはフロントの localStorage に保持。ペナルティ本体は OpenAI GPT-4o に長文プロンプトを1回投げる呼び出しとする。
- Q: `/api/punish` は OpenAI APIキーをどう入手するか？ → A: サーバー側の環境変数に所有者の OpenAI APIキーを1本保存しておき、ブラウザには一切キーを持たせない。
- Q: `/api/punish` は誰からの呼び出しのみを受け付けるか？ → A: QStash の署名検証（`Upstash-Signature` ヘッダー）を必須化し、検証失敗時は401で拒否する。
- Q: 解除ボタン押下時にQStashキャンセルAPIが失敗したらどうするか？ → A: キャンセル成功を確認するまで「解除完了」と表示せず、失敗時はエラーメッセージとリトライボタンを表示する。
- Q: アラーム時刻入力はどのタイムゾーンで解釈するか？ → A: ブラウザのローカルタイムゾーンで解釈し、Unixタイムスタンプ（絶対時刻）に変換して QStash へ渡す。

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - アラームをセットして時間内に解除する（優先度: P1）

ユーザーは就寝前にブラウザでアプリを開き、翌朝の起床時刻を指定してアラームをセットする。翌朝、設定時刻前に目を覚まし、ブラウザ上の「解除」ボタンを押す。ペナルティは発生せず、その日の料金請求もない。

**この優先度の理由**: これがアプリの正常系であり、「起きられた成功パス」として最もよく起こるべき体験。この体験が成立しないならアプリの価値（寝坊防止の動機付け）は得られない。

**独立テスト**: 時刻を短い先（例: 2分後）にセットし、その前に解除ボタンを押すことで完結して検証可能。ペナルティが発生しなかったこと（OpenAI APIが呼ばれなかったこと）をログ等で確認できればMVP価値を満たす。

**受入シナリオ**:

1. **Given** アラームが未設定の状態で **When** ユーザーが有効な未来の時刻を指定してセットボタンを押す **Then** アラーム時刻が保存され、解除ボタンが活性化したカウントダウン画面が表示される
2. **Given** アラームがセットされており現在時刻が設定時刻より前である **When** ユーザーが解除ボタンを押す **Then** アラームが解除済状態になり、ペナルティは発行されず、画面に解除完了メッセージが表示される
3. **Given** アラームが解除済 **When** その後設定時刻を過ぎる **Then** ペナルティ（OpenAI API自動実行）は発生しない

---

### ユーザーストーリー 2 - 解除し損ねてペナルティが実行される（優先度: P1）

ユーザーが寝過ごして設定時刻までに解除ボタンを押さなかった場合、設定時刻に達した時点で自動的にOpenAI APIが1回呼び出され、ユーザー自身のAPIキーに対して課金が発生する。ユーザーはペナルティが実行されたことを画面上で確認できる。

**この優先度の理由**: アプリの本質的な「罰金」機能。これが動かなければアプリの存在意義がなく、「寝坊しても何も起きない」という抜け穴になるためP1。

**独立テスト**: 2分後などの短いタイマーをセットし、解除せずに時刻を経過させて、OpenAI APIが1回呼ばれたこと・画面にペナルティ実行済表示が出ることを確認できる。

**受入シナリオ**:

1. **Given** アラームがセットされており解除されていない **When** 設定時刻に到達する **Then** ペナルティとしてOpenAI APIが1回だけ呼び出され、アラームはペナルティ執行済状態になる
2. **Given** ペナルティが執行された後 **When** ユーザーが画面を確認する **Then** 「ペナルティ執行済」と執行日時が表示される
3. **Given** ペナルティ執行済状態 **When** さらに時間が経過する **Then** 同じアラームに対して二重にペナルティが実行されることはない

---

### エッジケース

- アラーム発火予定時刻にブラウザが閉じられている／PCがスリープしている場合でも、Upstash QStash がサーバー側から `/api/punish` を呼び出すため、ペナルティは予定時刻どおり実行される。
- 設定済OpenAI APIキーが無効・残高不足などでAPI呼び出しが失敗した場合、ペナルティ実行はどう扱うか？（失敗してもアラーム側はペナルティ「試行済」として記録し、画面にエラーを表示する。罰金の目的は果たせないが二重実行は避ける。）
- ユーザーが現在時刻より過去の時刻をセットしようとした場合は、即座にバリデーションでエラーにする。
- ユーザーがアラーム発火前にページを再読み込みした場合、アラーム状態（設定時刻・解除済か）は保持されている必要がある。
- 同時に複数のアラームがセットできるか？ MVPではアクティブなアラームは1件のみとし、既存アラームがある場合は明示的に上書き確認を求める。
- 端末の時刻がずれている場合について、MVPではブラウザのローカル時刻をそのまま信頼し、変換後のUnixタイムスタンプをサーバー側（QStash）が絶対時刻として扱う。

## 要件 *(必須)*

### 機能要件

- **FR-001**: システムはユーザーがブラウザ上でアラーム発火時刻（年月日時分）をセットできる機能を提供しなければならない。入力値はブラウザのローカルタイムゾーンで解釈され、保存・送信時は Unix タイムスタンプ（絶対時刻）に変換すること。
- **FR-002**: システムは現在時刻より過去の時刻がセットされた場合、バリデーションエラーを返し保存しないこと。
- **FR-003**: OpenAI APIキーはサーバー側の環境変数として所有者があらかじめ設定しておくものとし、アプリのUI上には入力・保存・表示しないこと。
- **FR-004**: `/api/punish` はサーバー環境変数に OpenAI APIキーが設定されていない場合、ペナルティ実行を失敗扱いとしてログに記録すること。
- **FR-005**: システムはアラームの状態として少なくとも「設定済・未到達」「解除済」「ペナルティ執行済」「ペナルティ試行失敗」を区別して保持しなければならない。
- **FR-006**: ユーザーはアラームが設定時刻に達する前であれば任意のタイミングで解除ボタンを押してアラームを解除できること。
- **FR-007**: アラームセット時、システムは Upstash QStash に対して発火時刻に `/api/punish` エンドポイントを1回呼び出すスケジュールを登録し、返却されたスケジュールID をフロントエンドの localStorage に保存しなければならない。
- **FR-007a**: ユーザーが解除ボタンを押した場合、システムは localStorage に保存された QStash スケジュールID を使ってスケジュールをキャンセルし、キャンセルが成功（HTTP 2xx）として確認されるまでUI上で「解除完了」と表示してはならない。キャンセルAPIが失敗した場合はエラーメッセージとリトライボタンを表示すること。
- **FR-007b**: `/api/punish` エンドポイントは、呼び出された際にユーザーの OpenAI API キーを用いて OpenAI GPT-4o モデルに対し長文プロンプトを含むリクエストを1回送信し、課金を発生させること。
- **FR-007c**: `/api/punish` は Upstash QStash の署名ヘッダー（`Upstash-Signature`）を検証し、検証に失敗したリクエストは 401 で拒否し、OpenAI API を呼び出してはならない。
- **FR-008**: ペナルティ用プロンプトはMVPでは固定の長文でよいが、GPT-4o の課金対象として成立する十分な入力トークン数を持つこと。
- **FR-009**: システムは同一アラームに対してペナルティを二重実行してはならない（執行済・試行済フラグを用いて冪等性を確保すること）。
- **FR-010**: ペナルティAPI呼び出しが失敗した場合、システムはエラー内容を画面に表示し、状態を「ペナルティ試行失敗」として記録すること。
- **FR-011**: ブラウザを閉じたり再読み込みしたりしても、アクティブなアラームの状態および設定時刻は保持されなければならない。
- **FR-012**: 起動時に「過去に設定されたアラームで、現在時刻がその発火時刻を過ぎていて、まだ解除もペナルティ実行もされていない」ものを検出した場合、その時点で遅延実行としてペナルティを実行しなければならない。
- **FR-013**: 現在のアラーム状態、設定時刻までの残り時間、解除ボタン、現在ステータス（解除済／執行済など）はユーザーに常時可視であること。
- **FR-014**: MVPではアクティブなアラームは同時に1件に制限し、新規セット時に既存アラームがあれば上書き確認をユーザーに求めること。
- **FR-015**: 認証機能は提供しない。利用者は単一デバイス上の単一ユーザー（所有者）であることを前提とする。

### 主要エンティティ

- **アラーム（Alarm）**: ユーザーが設定した1件の起床目標。属性: 発火予定時刻、状態（未到達／解除済／執行済／試行失敗）、作成日時、解除日時、ペナルティ実行日時、最後のエラーメッセージ、QStashスケジュールID。
- **ユーザー設定（UserSettings）**: 単一ユーザーの設定。属性: OpenAI APIキー、ペナルティ実行時に使用するモデル／プロンプトなどの最小パラメータ。
- **ペナルティ実行ログ（PenaltyLog）**: 任意。ペナルティが実行された記録。属性: 実行日時、対象アラームID、成功／失敗、エラーメッセージ（もしあれば）。

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: ユーザーはアプリを開いてから30秒以内にアラーム時刻をセットし終えることができる。
- **SC-002**: アラームが設定時刻前に解除された場合、100%の確率でペナルティAPI呼び出しが発生しない。
- **SC-003**: アラームが解除されずに設定時刻を過ぎた場合、ブラウザのタブがアクティブである限り95%以上の確率で60秒以内にペナルティAPIが呼び出される。
- **SC-004**: 同一アラームに対してペナルティが二重実行される事象の発生率は0%である。
- **SC-005**: ブラウザを閉じて再度開いた後でも、未完了アラームの状態が100%復元される。
- **SC-006**: 初回セットアップ（APIキー登録 + 初回アラーム設定）を、アプリ未経験のユーザー（ただし本人）が2分以内に完了できる。

## 前提・仮定

- 利用者は1人（仕様の作者本人）であり、認証・マルチユーザー・データ共有は不要。
- 所有者は自身のOpenAI APIキー（課金有効）を保有しており、デプロイ時にサーバー環境変数として設定済であることを前提とする。
- ペナルティ実行時にブラウザタブがアクティブであり、JavaScriptが動作できる状態にあることを基本前提とする。バックグラウンドタブやスリープ中の完全な実行保証はMVPスコープ外とし、次回起動時の遅延実行で補う。
- アラーム状態の永続化にはブラウザ側のローカル永続ストレージ（例: localStorage 等に相当するもの）で十分とする。サーバー側の永続化は不要。
- モバイル対応は任意。少なくともデスクトップブラウザで動けばMVPは満たす。
- 音・通知の再生はペナルティの本質ではないため、MVPでは必須機能ではない（あると望ましい程度）。
- OpenAI APIキーはサーバー環境変数にのみ存在し、ブラウザ・ネットワーク上を流れない。
