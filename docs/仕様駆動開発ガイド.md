# 仕様駆動開発（Spec-Driven Development）の使い方ガイド

このリポジトリ（Agentic Workshop）は、AI コーディングアシスタント（Claude / Copilot / Gemini / Codex など）と一緒に **仕様駆動開発** を回すためのテンプレート・プロンプト・ワークフロー集です。本ドキュメントは、その使い方を日本語で整理したものです。

---

## 1. 全体像

仕様駆動開発は **「先に仕様を書き切ってから実装する」** アプローチです。本リポジトリでは以下 2 ブロックで構成されます。

| フェーズ | 内容 | 実行頻度 |
|---|---|---|
| **Phase 1** | PRD / TRD / UXS を書く → ロードマップ生成 → GitHub Issue をまとめて作成 | プロジェクトに **1 回だけ** |
| **Phase 2〜6** | Issue ごとに 実装 → レビュー → PR → マージ → 次 Issue 選定 | Issue ごとに **繰り返し** |

重要：Phase 1 は最初の 1 回きり。以降は Phase 2〜6 をループで回します。

---

## 2. 事前セットアップ

### 必要なもの
- エディタ（VS Code / Cursor など）
- GitHub アカウント
- AI コーディングアシスタント（サインイン済み）
- Git

### 手順（Mac 想定）

```bash
# 1. GitHub CLI
brew install gh
gh auth login
gh auth setup-git

# 2. uv（Python インストーラ）
curl -LsSf https://astral.sh/uv/install.sh | sh

# 3. Spec Kit をインストール
uv tool install specify-cli --force --from git+https://github.com/github/spec-kit.git
# Python が無ければ: uv python install 3.11

# 4. Spec Kit 初期化（Claude を使う場合）
specify init --here --force --ai claude --ai-skills
# 他: --ai copilot / --ai gemini / --ai codex

# 5. 確認
git --version && uv --version && specify --help
```

## 4. Phase 1：仕様を書く（1 回だけ）

新規プロジェクトの起点です。`docs/PRD-TRD-USX/` のプロンプトを順番に AI に渡して、3 種類の仕様書と Issue 群を生成します。

```
00-KICKOFF.md         ← オリエンテーション
      ↓
01-PRD-Prompt.md      → docs/PRD-[name].md に保存（プロダクト要件）
      ↓
02-TRD-Prompt.md      → docs/TRD-[name].md に保存（技術要件）
      ↓
03-USX-Prompt.md      → docs/UXS-[name].md に保存（UX 仕様）
      ↓
04-Roadmap.md         → ロードマップ作成 + GitHub Issue を一括生成
```

**成果物**：PRD 1 本 + TRD 1 本 + UXS 1 本 + GitHub Issue（Milestone 1 + Feature 4〜5 本）

各仕様の役割：
- **PRD (Product Requirements Document)** — 何を・なぜ作るか。受け入れ基準は **Gherkin（Given / When / Then）** で記述。
- **TRD (Technical Requirements Document)** — どう作るか（アーキテクチャ・技術選定）。
- **UXS (UX Specification)** — ユーザー体験・画面・インタラクション仕様。

Issue テンプレート（`.github/ISSUE_TEMPLATE/`）には **PRD-R# / TRD-R# / UXS-R#** といった仕様への参照欄があり、トレーサビリティを担保します。

## 5. Phase 2〜6：Issue ごとの実装ループ

Phase 1 で作成した Issue を 1 つずつ処理します。`docs/prompts/` 配下のプロンプトを順番に AI に渡していきます。

```
① 01p-start-issue.md            ── Phase 2：Issue 着手。specs/NNN-slug/ を作成し spec.md をコピー
        ↓
② plan.md & tasks.md を作成     ── Phase 3a：実装設計と作業分解
        ↓
③ 実装 & テスト                 ── Phase 3b〜4a
        ↓
④ 05p-validation-review.md      ── Phase 3：シニアエンジニア視点のレビュー
        ↓
⑤ 06p-validate-and-push.md      ── ローカル検証 → ブランチ push
        ↓
⑥ 07p-create-pr.md              ── Phase 4b：PR 作成
        ↓
⑦ 08p-review-pr.md              ── Phase 4c：レビューコメント対応
        ↓
⑧ 09p-merge-branch.md           ── Phase 5：main に --no-ff マージ
        ↓
⑨ 10p-post-implementation-review.md ── Issue クローズ & 振り返り
        ↓
⑩ next-issue-handoff.md         ── Phase 6：次 Issue を選定
        ↓
      ループ → ① へ戻る（Phase 1 には戻らない）
```

補助プロンプト：
- **spec-review-checklist.md** — Issue 単位の `spec.md` / `plan.md` の網羅性チェック
- **context-pack.md** — コンテキスト読み込みポリシー（一度読んだら使い回す）

## 6. 使い始め方（クイックスタート）

### 新規プロジェクトの場合
1. `AGENTS.md` を読む（1 分）
2. `docs/contributing/AGENT-WORKFLOW.md` をロード（5 分）
3. **Phase 1** を実行：
   - `docs/PRD-TRD-USX/00-KICKOFF.md` から開始
   - PRD → TRD → UXS を書いて `docs/` に保存
   - `04-Roadmap.md` で Issue 一括生成
4. **Phase 2〜6** を Issue ごとに繰り返し

### 既存 Issue に着手する場合
1. `docs/contributing/AGENT-WORKFLOW.md` の Phase 2 以降を読む
2. `docs/prompts/01p-start-issue.md` を AI に渡して開始
3. Phase 2〜6 ループを回す

---

## 7. 本ワークフローの特徴

- **スペックファースト**：Issue を切る前に PRD / TRD / UXS を書き切る
- **Phase 1 は 1 回のみ**：仕様作成は `04-Roadmap.md` で完結
- **Phase 2〜6 は繰り返し**：`next-issue-handoff` で次 Issue に自動遷移
- **言語・フレームワーク非依存**：JS / Python / Rust / Go など何でも OK。`make` / `npm` / `cargo` などの汎用ビルドコマンドで記述
- **外部スクリプト不要**：`git` と `gh` CLI のみで完結
- **AI フレンドリー**：全プロンプトがエージェント実行を前提に設計

## まとめ

1. **最初に 1 回**：`docs/PRD-TRD-USX/` のプロンプトで PRD / TRD / UXS と Issue を生成
2. **Issue ごとに繰り返し**：`docs/prompts/` のプロンプトを順番に実行して実装 → PR → マージ
3. **ループの司令塔**：`docs/contributing/AGENT-WORKFLOW.md` と `AGENTS.md` を常にロード

これが本リポジトリにおける仕様駆動開発の回し方です。
